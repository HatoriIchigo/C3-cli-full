# ToDoアプリケーション 要件定義書

## はじめに

### このドキュメントの位置づけ
本ドキュメントは、ToDoアプリケーション開発プロジェクトにおける要件定義書です。開発チームが機能実装を行う際の基盤となる要件を定義し、プロジェクト関係者の共通理解を促進します。

### このアプリケーション導入の目的
タスク処理を円滑にするためのToDoアプリケーションを構築し、個人のタスク管理効率を向上させることを目的とします。ユーザが日常的なタスクを効率的に管理・追跡できる環境を提供します。

### システム概要
AWS基盤上でReact（フロントエンド）、Node.js + Express（バックエンド）、MySQL（データベース）を用いたWebベースのToDoアプリケーションです。階層構造を持つタスク管理機能とユーザ認証機能を提供し、個人開発としてコストパフォーマンスを優先した設計とします。

## システム要件

### 1. 機能要件
- ユーザ認証機能（ユーザ名・パスワード認証）
- 階層タスク管理機能（親タスク・子タスクの2階層構造）
- タスクのCRUD操作機能
- ユーザ専用のタスク表示機能

### 2. 運用・保守要件
- 手動でのEC2インスタンス起動・停止による運用
- コスト最適化を優先した運用体制
- 個人開発レベルでの保守性確保

### 3. 監視要件
- 基本的なアプリケーション稼働監視
- データベース接続状況の監視
- エラーログの収集・確認機能

### 4. 可用性
- 日中の利用時間帯での基本的な可用性確保
- サービス停止時の影響を最小限に抑制

### 5. 性能・拡張性
- 単一ユーザでの快適な操作性確保
- 基本的なレスポンス性能の維持
- 将来的な機能拡張への対応可能性

### 6. 検証環境
- 開発環境での基本的な動作確認
- 本番環境での最低限の動作検証

### 7. その他要件
- コストパフォーマンスを最優先とした設計
- 個人開発規模に適した技術選択

## 機能要件

### アプリケーション機能要件

#### OS/ソフトウェア
- AWS EC2インスタンス（フロントエンド・バックエンド各1台）
- Amazon RDS for MySQL
- セキュリティグループによる適切なアクセス制御

#### フロントエンド機能
- **技術スタック**: React
- **要件**:
  - 概要: ユーザフレンドリーなWeb UIの提供
  - 可用性: ブラウザベースでのアクセス性確保
  - 性能・拡張性: シングルページアプリケーションとしての軽快な動作

#### バックエンド機能
- **技術スタック**: Node.js + Express
- **要件**:
  - 概要: RESTful APIによるデータ処理とビジネスロジックの実装
  - 可用性: フロントエンドからの安定したAPI提供
  - 性能・拡張性: 効率的なデータベースアクセスと処理性能の確保

#### データベース機能
- **技術スタック**: Amazon RDS for MySQL
- **要件**:
  - 概要: ユーザ情報とタスクデータの永続化
  - 可用性: データの整合性と耐久性確保
  - 性能・拡張性: 適切なインデックス設計による検索性能最適化

## 非機能要件

### 1. 可用性
- 手動でのインスタンス管理による計画的な停止時間設定
- 必要時のサービス復旧手順の明確化
- データバックアップによるデータ保護

### 2. 性能
- Webページの読み込み時間: 5秒以内
- API応答時間: 3秒以内
- データベースクエリ応答時間: 2秒以内

### 3. 拡張性
- 将来的な機能追加への対応可能な設計
- モジュール化されたコード構造
- 設定変更による環境切り替え対応

### 4. 監視
- アプリケーションログの出力と確認
- データベース接続エラーの検知
- 基本的なシステムリソース監視

### 5. 運用・保守
- 手動でのサーバ起動・停止運用
- 定期的なデータバックアップ
- 障害時の復旧手順書整備

### 6. セキュリティ
- ユーザ認証による適切なアクセス制御
- パスワードの適切なハッシュ化
- SQLインジェクション対策
- XSS対策の実装
- HTTPS通信の採用

## ユースケース/シナリオ

### 基本的な利用シナリオ
1. ユーザログイン
2. 親タスクの作成・編集・削除
3. 子タスクの作成・編集・削除・完了
4. タスク一覧の表示・フィルタリング
5. ログアウト

### タスク管理シナリオ
- ユーザは大きなプロジェクト（親タスク）を作成できる
- 各プロジェクト内で具体的な作業項目（子タスク）を管理できる
- 子タスクの完了状況から親タスクの進捗を把握できる

## 制約条件

### 技術的制約
- AWS無料利用枠の活用によるコスト制約
- 個人開発での工数制約
- 既存技術スタックの活用による学習コスト削減

### 運用制約
- 手動でのサーバ管理による運用時間制約
- 24時間稼働ではなく必要時のみの稼働
- 個人での保守・運用体制

### 予算制約
- AWS利用料金の最小化
- オープンソース技術の積極活用
- 追加ライセンス費用の回避
